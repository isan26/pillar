# V1 Conversation Debug Logging

## Summary

Adopt a small local observability layer for the current console app. The goal is to save conversations, turns, model usage, errors, and debug artifacts in a gitignored folder without doing a full framework rewrite yet.

## Key Changes

- Add `debug/` to `.gitignore` and store local run artifacts under `debug/conversations/{timestamp_utc}-{session_id}/`.
- Add simple IDs:
  - `session_id`: short UUID per app launch.
  - `turn_id`: `turn_001`, `turn_002`, etc.
  - `run_id`: `run_001`, `run_002`, etc.
- Save these files per session:
  - `session.json`: session metadata, model, start/end timestamps.
  - `messages.json`: user-visible transcript only.
  - `turns.json`: one record per user input lifecycle.
  - `runs.json`: one record per model call.
  - `events.jsonl`: append-only event stream.
  - `conversation.md`: readable transcript.
- Prefer `conversation.md` inside the session folder over the current root-level `conversation.txt`.
- Do not add `app/`, repositories, database interfaces, RAG folders, tools folders, or raw retrieval artifacts yet.

## Runtime Behavior

- On startup, print the configured model and debug folder:

```text
Model: {MODEL}
Debug folder: debug/conversations/{timestamp}-{session_id}
```

- For the startup greeting, log `app_greeting_printed` if it is static. Only create an LLM run with purpose `greeting` if the greeting is generated by the model.
- For each user input, create events for:
  - `turn_started`
  - `user_message_saved`
  - `llm_run_started`
  - `llm_run_completed` or `llm_run_failed`
  - `assistant_message_saved`
  - `turn_completed` or `turn_failed`
- On model errors, save the error to `runs.json` and `events.jsonl`, then print a short friendly message including the configured `MODEL`.
- If a model error is shown to the user as a friendly fallback message, save that assistant message with `source: "local_error_fallback"`. If no fallback is shown as chat output, do not create an `assistant_message_saved` event.
- Do not log `.env`, API keys, authorization headers, cookies, or secret-looking fields.

## Usage Tracking

- After each `agent.invoke(...)`, inspect the final assistant message:

```python
ai_message.usage_metadata
ai_message.response_metadata
```

- Save per-run metadata:

```json
{
  "session_id": "a1b2c3d4",
  "run_id": "run_001",
  "turn_id": "turn_001",
  "provider": "openai",
  "model": "openai:gpt-5.4",
  "purpose": "chat_response",
  "status": "success",
  "started_at": "2026-05-25T12:00:00Z",
  "completed_at": "2026-05-25T12:00:01Z",
  "input_tokens": null,
  "output_tokens": null,
  "total_tokens": null,
  "latency_ms": 0,
  "estimated_cost_usd": null,
  "raw_usage_metadata": {},
  "raw_response_metadata": {},
  "error_type": null,
  "error_message": null
}
```

- Add optional env vars:

```env
DEBUG_DIR=debug
MODEL_INPUT_PRICE_PER_1M_TOKENS=
MODEL_OUTPUT_PRICE_PER_1M_TOKENS=
```

- If prices are unset, record token counts and set estimated cost to `null`.
- If prices are set, calculate estimated cost from input/output token totals.
- These prices apply to the currently configured `MODEL`; leave unset if unknown.
- Use `null` for unavailable token counts. Do not use `0` unless the provider explicitly reports zero.
- Store normalized token fields plus the raw LangChain `usage_metadata` and `response_metadata` dictionaries from the final AI message.

## Implementation Shape

- Keep everything in `agent.py` for this step, but organize with small functions:

```python
create_session()
next_turn_id()
next_run_id()
append_event(...)
save_json(...)
save_jsonl(...)
save_messages()
save_run(...)
extract_usage(...)
calculate_cost(...)
sanitize_for_logging(...)
```

- Use standard library only: `json`, `time`, `uuid`, `datetime`, `pathlib`.
- Use UTF-8 for all debug files.
- Use `pathlib.Path` for cross-platform paths.
- Include `session_id` in every saved message, turn, run, and event record, even though files live inside a session folder.
- Include `started_at`, `completed_at`, and `latency_ms` for turns and runs.
- Apply recursive log sanitization before writing metadata or errors. Redact secret-like keys such as `api_key`, `apikey`, `authorization`, `bearer`, `cookie`, `password`, `secret`, `access_token`, `refresh_token`, `client_secret`, and keys named `token` or ending in `_token`/`-token`, without redacting token-count fields like `input_tokens`.
- Do not introduce a `ConversationStore` class yet; that abstraction becomes useful once the project has multiple files or a second storage backend.

## Test Plan

- Run `python -m py_compile agent.py`.
- Start the app and verify a session folder is created.
- Send one normal message and verify `messages.json`, `turns.json`, `runs.json`, `events.jsonl`, and `conversation.md` update.
- End the conversation and verify final session metadata is saved.
- Temporarily set an invalid `MODEL` and verify the error is logged without exposing secrets.
- Confirm `debug/` is ignored by git.

## Assumptions

- Local JSON/JSONL is enough for now; database comes later.
- Exact low-level model API request/response logging is out of scope for v1. For now, log the reliable normalized data available from LangChain: messages, model, token usage, latency, and errors.
- For learning, prioritize clear local traceability over production-grade architecture.
- Retrievals, tool-call trace linking, routing decisions, prompt versions, memory events, and guardrails are future phases.
